## 摘要
AI 代码智能体在处理孤立任务时表现出色，但在处理复杂的、涉及数十个模块间关系的跨文件软件工程任务时却面临挑战。我们假设这些失败源于智能体在探索代码库的过程中，无法构建、维护和更新连贯的架构信念（architectural beliefs）。我们提出了代码空间理论（Theory of Code Space, ToCS），这是一个评估该能力的基准测试。它将智能体置于部分可观测的程序化生成代码库中，要求它们针对模块依赖、横切不变性（cross-cutting invariants）和设计意图构建结构化的信念状态。该框架具有以下特点：(1) 一个程序化代码库生成器，可生成中等复杂度的 Python 项目，包含反映不同发现方法的四种类型化边类别——从语法导入到配置驱动的动态连线——并植入了架构约束和经过验证的基准真相（ground truth）；(2) 一个部分可观测性控制装置，智能体在受限的预算下进行探索；(3) 通过结构化 JSON 进行周期性的信念探测，生成架构理解的时间序列。我们将空间推理基准测试中的“主动-被动差距”（Active-Passive Gap）分解为选择和决策组件，并引入“架构约束发现”作为代码特有的评估维度。使用来自三个供应商的四个基于规则的基准模型和五个前沿大语言模型（LLM）智能体进行的初步实验验证了该基准的区分能力：各方法的性能跨度很大（F1 分数从 0.129 到 0.646），LLM 智能体能够发现所有基准模型都无法识别的语义边类型，但较弱的模型得分低于简单的启发式方法——这揭示了信念外部化（即忠实地将内部理解序列化为结构化 JSON）本身就是一种非平凡的能力，也是信念探测基准测试中的一阶混杂因素。开源工具包：https://github.com/che-shr-cat/tocs

## Abstract
AI code agents excel at isolated tasks yet struggle with complex, multi-file software engineering requiring understanding of how dozens of modules relate. We hypothesize these failures stem from inability to construct, maintain, and update coherent architectural beliefs during codebase exploration. We introduce Theory of Code Space (ToCS), a benchmark that evaluates this capability by placing agents in procedurally generated codebases under partial observability, requiring them to build structured belief states over module dependencies, cross-cutting invariants, and design intent. The framework features: (1) a procedural codebase generator producing medium-complexity Python projects with four typed edge categories reflecting different discovery methods -- from syntactic imports to config-driven dynamic wiring -- with planted architectural constraints and verified ground truth; (2) a partial observability harness where agents explore under a budget; and (3) periodic belief probing via structured JSON, producing a time-series of architectural understanding. We decompose the Active-Passive Gap from spatial reasoning benchmarks into selection and decision components, and introduce Architectural Constraint Discovery as a code-specific evaluation dimension. Preliminary experiments with four rule-based baselines and five frontier LLM agents from three providers validate discriminative power: methods span a wide performance range (F1 from 0.129 to 0.646), LLM agents discover semantic edge types invisible to all baselines, yet weaker models score below simple heuristics -- revealing that belief externalization, faithfully serializing internal understanding into structured JSON, is itself a non-trivial capability and a first-order confounder in belief-probing benchmarks. ### 2. 论文提出的方法论
ToCS 框架包含三个关键设计选择：
*   **程序化代码库生成器**：生成中等复杂度的 Python 项目（Pipeline 架构）。它定义了四种类型的依赖边：
    1.  `IMPORTS`：静态导入（最易发现）。
    2.  `CALLS_API`：跨模块函数调用。
    3.  `DATA_FLOWS_TO`：跨模块数据流向。
    4.  `REGISTRY_WIRES`：基于配置文件的动态加载（最难发现）。
*   **部分可观测性环境**：智能体无法一次性看到全部代码，必须在有限预算（如 20 次操作）内通过 `LIST`、`OPEN`、`SEARCH`、`INSPECT` 等工具主动探索。
*   **周期性信念探测（Probing）**：每隔 $K$ 步，系统会要求智能体以结构化 JSON 形式外部化其当前的架构理解（包括组件、依赖边、不变性约束和不确定性）。
*   **主动-被动差距（APG）分解**：将性能差距分解为“选择组件”（决定看哪些文件）和“决策组件”（如何处理观察到的信息）。

    *   **Dependency F1**：评估依赖图识别的准确性。
    *   **Invariant F1**：评估对植入的架构约束（如“禁止直接依赖”）的发现能力。

### 6. 主要结论与发现
*   **LLM 优于规则模型**：最强的智能体（GPT-5.3-Codex 和 Claude Sonnet 4.6）在依赖识别 F1 上超过了所有基于规则的基准。
*   **语义理解能力**：LLM 能够发现规则模型无法识别的 `DATA_FLOWS_TO` 和 `REGISTRY_WIRES` 等深层语义边。
*   **信念外部化瓶颈**：弱模型表现不佳并非因为没“看到”代码，而是无法将理解准确地序列化为 JSON。
*   **信念崩溃现象**：观察到某些模型（如 Gemini 2.5 Pro）存在“灾难性遗忘”，在探索过程中会突然丢失之前已正确识别的架构信息，而较小的模型（Gemini 2.5 Flash）反而表现出更稳定的单调知识积累。
*   **提示词敏感性**：改进探测提示词（提供更明确的语义定义）能显著提升架构约束发现的得分。

### 7. 优点
*   **诊断性强**：不同于以往只看最终代码产出的基准，ToCS 专注于评估智能体的**中间认知过程**。
*   **任务设计科学**：引入了空间推理中的 APG 概念，并针对软件工程特有的“架构约束”设计了评估维度。
*   **防泄露**：程序化生成的代码库确保了模型无法通过预训练记忆来“作弊”。

### 8. 不足与局限
*   **架构单一**：目前仅支持 Pipeline 架构，无法代表微服务、事件驱动等其他复杂架构。
*   **语言限制**：目前仅限于 Python。
*   **缺乏命名先验**：为了隔离推理能力，生成器使用了中性文件名（如 `mod_a.py`），这剥离了现实中开发者依赖的语义线索（如 `auth_service.py`）。
*   **外部化偏差**：JSON 探测可能无法完全反映模型的内部状态，模型可能“知道”但“写不出来”。
