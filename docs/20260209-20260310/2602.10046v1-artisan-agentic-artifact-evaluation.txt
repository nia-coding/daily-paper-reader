# Artisan: Agentic Artifact Evaluation 

DOEHYUN BAEK, University of Stuttgart, Germany 

MICHAEL PRADEL, CISPA Helmholtz Center for Information Security, Germany Artifact evaluation has become standard practice in the software engineering community to ensure the reproducibility of research results. However, the current manual process is labor-intensive, and hence, done only as a one-time assessment for a subset of all papers. To support the artifact evaluation effort, we present Artisan, an automated LLM agent for reproducing research results given a paper and its artifact. The approach is enabled by two key contributions: First, we frame the reproduction problem as a code generation task where the goal is to generate a reproduction script that, when executed, reproduces the results reported in a paper. Unlike prior work on automatically reproducing research results, this formulation allows for running the script independently of the agent and for assessing the reproduction process at a fine-grained level. Second, we design automated judging mechanism that guides the agent toward the expected results without revealing them and that prevent trivial solutions, such as simply copying checked-in results. To evaluate Artisan, we introduce Artisan-Bench, the first benchmark assessing the ability to generate reproduction scripts and the first benchmark for automated artifact evaluation in software engineering. Artisan-Bench comprises 60 tasks derived from 23 software engineering papers, covering different research areas and programming languages. We validate all tasks in Artisan-Bench for reproducibility to ensure that the tasks are feasible. Our experiments show that Artisan is effective, producing 44/60 reproduction scripts and outperforming the best available baseline, a vanilla LLM agent (mini-swe-agent), by 3.14 × in terms of reproduction scripts generated while taking $0.45 and 48 minutes, on average per task. Beyond validating reproducibility, Artisan helped uncover 20 new errors in either the paper or artifact, further demonstrating the practical utility of our approach.

1 INTRODUCTION 

Since the pilot of artifact evaluation at ESEC/FSE 2011 [ 26 ], artifact evaluation has become standard practice in the software engineering community, subsequently being adopted in the programming languages [ 27 ], security [ 35 ], and database communities [ 7]. One of its main goals is reproducibility ,defined as the ability of a different team to obtain similar experimental outcomes using the same experimental setup [ 6]. Reproducibility is generally regarded by participants of artifact evaluation committees as one of the most important goals of artifact evaluation [ 21 ]. In addition, studies report that artifact evaluation leads to more available, maintained, and documented artifacts [46]. Despite these benefits, artifact evaluation in its current manual form is notoriously labor-intensive. Reports indicate that evaluating a single artifact can take eight to ten hours, i.e., often longer than reviewing the paper itself [ 20 ]. The current process yields the Artifacts Evaluated badge as its out-come [ 6], which provides limited signal regarding reproducibility: First, the badge is coarse-grained, i.e., even if an artifact has the badge, some parts of the paper may not be reproducible. In fact, recent work on the reproducibility of artifacts [ 33 ] indicates that, among 100 ICSE papers with artifacts that were studied, only 14 reproduce the original findings. Motivated by this limitation, we pose the question: What if we task LLM agents with generating code that reproduces the results? Automated artifact evaluation powered by code-generating agents would offer the following benefits: (1) It significantly reduces manual effort. (2) It scales to a much larger number of papers and artifacts. (3) It provides a fine-grained assessment of specific research results. (4) It can be run before the submission of a paper. (5) It can be run continually in the future. This paper presents Artisan, the first automated artifact evaluation approach utilizing an LLM agent tasked with code generation. Given a table with numerical results from a research paper, Artisan autonomously takes actions to reproduce the results. The approach is enabled by two key contributions: First, we formulate artifact reproduction as a code generation task, where the agent generates a reproduction script that independently reproduces the paper’s reported results based on the code and data given in the artifact. Second, we introduce automated judging mechanism that steers the agent toward correct results without exposing them, while preventing shortcuts like copying pre-computed results. Artisan is closely related to, but differs in several ways from, recent work in ML and NLP on automated research reproduction, summarized in Table 1. These approaches can be classified based on the agents’ output. Both rating-based and output-based agents, when unable to reproduce the results, lack credible evidence that the error lies with the paper or the artifact. Given that agents are an emerging technique, failures are typically attributed to the agents. To our knowledge, there are no reports of errors in artifacts identified by agents during reproduction attempts. Artisan differs by having the agent generate a reproduction script, which is executable independently of agents. When the paper’s results disagree with those produced by running the reproduction script, which we term paper-artifact inconsistencies , it acts as a credible evidence that the error is in the side of paper or artifact. To evaluate Artisan, we develop Artisan-Bench, the first benchmark assessing LLM agents’ capabilities to generate reproduction scripts. Artisan-Bench comprises 60 tasks derived from 23 software engineering papers. We manually validate all tasks in Artisan-Bench for reproducibility, and the benchmark covers diverse software engineering techniques and programming languages not covered by the existing benchmarks. Evaluating Artisan on the benchmark shows that Artisan is effective, producing 44/60 reproduction scripts and outperforming the best available baseline, a vanilla agent (mini-swe-agent), by 3.14 ×. The approach is also efficient, taking an average of $0.45 and 48 minutes per task. Beyond validating reproducibility, Artisan also helped identify 20 previously unknown inconsistencies between papers and their artifacts, further demonstrating the practical utility of our approach.

2 APPROACH 

We first define the problem we address (Section 2.1) and give a brief overview of the approach (Section 2.2). Then, we present in detail the preparation mechanisms (Section 2.3), tools (Section 2.4), and judging mechanism (Section 2.5) that enable our approach. 

2.1 Problem Definition 

We address the following problem: given a research paper 𝑝 , a table 𝑡 with numerical results from 

𝑝 , and an artifact URL 𝑢 , automatically generate a reproduction script 𝑠 that reproduces 𝑡 using the artifact. The reproduction script 𝑠 should contain all logic needed to reproduce the table from scratch, from running the artifact to formatting outputs into the final table. The reproduction script 

𝑠 may reuse existing artifact code or replace it when necessary. If the artifact already provides a script that fully regenerates the target table, then 𝑠 can simply call it. 1 If reproduction requires multiple scripts, then 𝑠 should run them in the right order. If provided scripts are non-functional (e.g., an obsolete Dockerfile ), 𝑠 may fix them, so the table can still be reproduced. Our approach enforces two properties. First, the approach must produce a reproduction script that runs without an agent. Prior work [ 10 , 22 , 24 , 39 ] employs LLM agents to reproduce results directly and output a rating or numbers. Users of these agents must blindly trust the agent output or inspect long agent trajectories. Generating a reproduction script avoids this: users can run and audit the generated script instead of the agent’s trajectory. Second, the generated script should reproduce results as fully as possible within the time budget. Prior work [ 10 , 22 , 24 , 39 ] focuses only on whether outputs match, not how they are produced. In practice, there are multiple ways to reproduce results with different levels of completeness. Ignoring the method of reproduction can

2.2 Overview 

Figure 1 gives an overview of Artisan. Given a paper, a table, and an artifact URL, Artisan generates a reproduction script that reproduces the table from scratch using the artifact. Artisan is an LLM agent, i.e., an LLM-based system that uses tools to autonomously interact with the environment to achieve a given goal [ 11 ]. It follows a five-step workflow: (1) Download the artifact, (2) Read the README, (3) Find relevant commands (4) Execute them, and (5) Create a reproduction script. To implement the workflow, Artisan first prepares inputs (Section 2.3) by obfuscating table digits with question marks to avoid exposing expected results, and by downloading the artifact with the given URL. The agent uses two tools (Section 2.4): a general bash tool for standard CLI operations, and a task-specific format tool to transform execution outputs into the expected table format. Once the agent submits a script, a two-tier judge (Section 2.5) runs it to verify the output matches the expected results, then checks the reproduction method is meaningful rather than copying the checked-in results. In case the submitted script fails either of the two judging steps, the agent is given corresponding feedback and continues its iteration. Otherwise, Artisan outputs the submitted script as the final reproduction script. The high-level control-flow of Artisan is summarized in Algorithm 1. First, Artisan obfuscates the input table 𝑡 (line 4) and downloads the artifact (line 5). The results of the preparation mechanisms are used to construct a user prompt to be handed to the LLM (line 6). If the command invoked by the LLM is a tool call, one of our tools is executed (line 10). In case of the special “submit” command, the algorithm checks the output of the submitted script (line 13), and if successful, also the method of reproduction (line 16). If the submitted script passes both judging steps, Artisan outputs it as the final reproduction script (line 19). Otherwise, the algorithm resumes the agentic loop.

To address this challenge, our preparation mechanism obfuscates the numerical values of the expected table before giving it to the agent. Figure 3 shows an example of the obfuscation mechanism. 2.3.2 Artifact Download. A typical artifact URL points to a repository hosting the artifact, such as Zenodo, Figshare, or GitHub. Given such a URL, the agent must first download the artifact file before it can proceed with reproducing the results. However, we found that agents often struggle to find the correct way to download the actual artifact file given the artifact URL. Figure 2b shows an example, where the agent is given the artifact’s Zenodo URL and presumes the artifact is downloadable through a canonical file-download endpoint. However, this endpoint does not exist, and the agent downloads an HTML file containing “Page not found”. The agent, assuming a successful download, attempts to unzip the file and realizes it is not a valid ZIP archive. To address this challenge, we provide a download mechanism, which the agent can invoke with only the URL of the research artifact. Given this URL, the download mechanism first queries the repository to obtain the list of files contained in the artifact. It then downloads each file, caching any file that has not been retrieved before. Finally, it locates the README file by searching for files whose names begin with “README” among all downloaded contents, including those stored inside ZIP or TAR archives. Although simple, this strategy effectively saves the agent several steps that would otherwise be spent reasoning about repository-specific APIs. The download mechanism currently supports Zenodo, Figshare, and GitHub. 

2.4 Tools 

We provide the bash tool (Section 2.4.1) for general CLI interaction and the format tool (Section 2.4.2) developed specifically for the challenge of automated artifact evaluation. 

2.4.1 Bash Tool. We utilize the bash tool of mini-swe-agent [ 42 ] to handle generic CLI usage tasks, encompassing reading file, writing file, and running commands. We found that the bash tool is enough to handle most general CLI usage that arises in the task of automated artifact evaluation. The vanilla mini-swe-agent we compare against in the evaluation contains only the bash tool. 

2.4.2 Format Tool. Even after an agent successfully reproduces results by running a sequence of commands, the outputs rarely match the exact format shown in the paper. 2.5 Judging Mechanism 

A key component of Artisan is to automatically judge whether the reproduction script generates correct output with appropriate method. We present a two-tier judging mechanism that combines an execution-based output judge (Section 2.5.1) and an LLM-based method judge (Section 2.5.2). 

2.5.1 Execution-Based Output Judge. Previous work on rating-based agents [ 22 , 24 ] and output-based agents [ 10 , 39 ] for reproducibility asked agents to yield simple numerical results as a final output. However, as Artisan asks the agents to output reproduction script, there are additional failure modes that arise when the script triggers syntax or runtime errors. Often, the agents successfully execute commands to reproduce the research results, but there is some fault in the submitted script, which can be often addressed by giving an error message as a feedback. Figure 2d shows an example trajectory of an agent facing this challenge. The agent writes a reproduction script that utilizes awk to extract research results from an execution output, but uses awk syntax that is invalid with the standard POSIX awk. Unfortunately, the agent submits the script without running it and signals completion. To address this challenge, the output judge runs the submitted script in a fresh container envi-ronment and classifies the result into four categories: 

• Static error : The submitted script is missing from the expected path or fails to run because of a syntax error. 

• Runtime error : The submitted script runs but produces a runtime error. 

• Mismatched results : The submitted script runs to completion without any error, but the results differ from the expected values. 

• Success : The submitted script runs to completion without any error, and the results match the expected values. In the first three cases, Artisan gives corresponding feedback to the agent, allowing it to continue its iteration. Importantly, in the case of mismatched results, the output judge reveals parts of the result the agent got right, but not the parts it got wrong. Figure 4 gives an example of this feedback

2.5.2 LLM-Based Method Judge. While developing Artisan, we encountered many cases where the LLM agent produces a reproduction script that yields the correct results for the wrong reasons. A typical scenario is when the agent hard-codes the expected results inside of the reproduction script or simply copies them from pre-computed results that are part of the artifact. Figure 2e shows an example trajectory of an LLM agent facing this challenge. Upon discovering that the artifact contains the research results of Table 1 checked in, the agent writes a reproduction script that utilizes these results and signals completion. As the overall goal is to reproduce the research results, such “short-cuts” are undesirable. To address this challenge, if the submitted script passes the output judge, the method judge, an LLM-as-judge evaluating the method of reproduction, is invoked. The method judge classifies the reproduction method into three categories: 

• Copied results : The submitted script just copies the expected values from the checked-in results or the paper without any meaningful reproduction. An example is shown in Figure 5a, where the script copies checked-in results. 

• Last-mile reproduction : The submitted script performs a lightweight reproduction, skipping some time-consuming steps. An example is shown in Figure 5b, where the script uses checked-in raw data and processes it to generate results. Although this falls short of full reproduction, we judge this to be a valid reproduction. This is because the full reproduction of some experiments may take hundreds of hours (as in the example), and verifying that the analysis of the raw data matches the paper’s results is still valuable. 

• Full reproduction : The submitted script performs the most complete reproduction possible. This is the ideal case, especially if it can be achieved within a reasonable time. An example is shown in Figure 5c, where the script performs the full reproduction, from raw data generation to analysis, on one example. If the method judge classifies the submitted script as either last-mile reproduction or full repro-duction, Artisan accepts the submitted script as the final reproduction script. In contrast, scripts classified as copied-results are rejected, and the approach yields control back to the agent, asking it to find an alternative, more complete reproduction method.

3 BENCHMARK 

In this section, we present the methodology used to construct Artisan-Bench. We first state the scope and requirements we target (Section 3.1), then describe the selection of papers (Section 3.2) and tasks (Section 3.3). Finally, we explain the ground truth scripts we prepare for each task (Section 3.4). 

3.1 Scope and Requirements 

Our construction of Artisan-Bench is guided by two requirements: (1) We require a manual val-idation of the reproducibility for all tasks. This requirement is motivated by SWE-Bench [ 25 ], where subsequent efforts, such as SWE-Bench Verified [ 16 ] and Wang et al. [ 45 ], highlight the importance of manual validation to avoid overestimating or underestimating agent capabilities. (2) The benchmark should cover a wide range of software engineering areas and programming languages. This requirement is motivated by the fact that existing benchmarks [ 10 , 24 , 39 ] focus on machine learning and natural language processing techniques implemented in Python, R, and Stata, and no benchmark currently exists for software engineering research reproduction. 

3.2 Paper Selection 

Guided by these requirements, we curate a set of 23 papers following the process below. (1) Papers with badges. We gather 114 papers from four flagship software engineering venues (ICSE, FSE, ASE, ISSTA) in 2024 with ACM Artifacts Available and Artifacts Evaluated—Reusable 

badges [ 6]. We manually locate links to the accompanying artifacts. For FSE and ISSTA, the ACM Digital Library lists artifact links. For ICSE and ASE, we prioritize links to the artifacts found in the paper. If we fail to find such links in the paper, we search the web for the paper title. (2) Docker. We exclude 42 papers whose artifacts are not packaged using Docker, which yields 72 remaining papers. This reflects the fact that Docker and related virtualization technologies are now standard in artifact evaluation [ 1 –3 ]. To implement this, we case-insensitively search Markdown and PDF files for the term “docker”. (3) Non-public APIs. We exclude nine papers that rely on non-public APIs, yielding 63 remaining papers. To implement this, we search Python files and Jupyter notebooks for “openai” or “anthropic”. We also search Markdown files for “etherscan”. This excludes seven papers using OpenAI APIs and two papers using Etherscan APIs. (4) Specialized hardware. We exclude 19 papers that require GPUs or other specialized hardware, yielding 44 remaining papers. This is to maintain accessibility across hardware setups, similar to previous work [ 10 ]. We search for common GPU keywords (e.g., cuda , cudnn , tensorflow ,

> nvidia-smi

, gpus ) in source code files. (5) Reproducibility budget. We exclude 21 papers to enforce an eight-hour reproducibility budget, yielding the final set of 23 papers. We first filter out 12 papers deemed infeasible by searching for the keyword “hour”; these explicitly mention experiment times exceeding the limit (four take >12 hours, seven >24 hours, and one >125 hours). We then try to manually reproduce the remaining 32 papers. We exclude nine papers that we could not reproduce any of the tables within the budget. These failures stemmed from missing datasets (2x), missing raw data (2x), missing generation scripts (2x), non-deterministic outputs (2x), or an inaccessible Docker image (1x). For the papers we could manually reproduce some of the tables, we include the paper in the benchmark but exclude the tables we fail to reproduce. The selected 23 papers cover a wide range of software engineering areas and programming languages. Specifically, we classify five papers as empirical studies, five as static analysis, two as

Notably, there is a possible under-representation of AI4SE and SE4AI techniques, which is mainly due to our exclusion of papers using non-public APIs and GPUs. Nevertheless, one paper [ 28 ] makes use of classical machine learning techniques (Random Forest and XGBoost) and uses neither non-public APIs nor GPUs. In terms of programming languages, our selection covers Python (13x), Java (4x), Rust (2x), Scala (2x), OCaml (1x), and Bash (1x). 

3.3 Task Selection 

From the 23 selected papers, we curate 60 tasks. Starting from all 89 tables contained in the 23 papers, we exclude 21 non-result tables, which contain descriptions of evaluation setups (14x) or the approach (7x). We also exclude six tables that we fail to reproduce during our paper selection process and one table that requires 400 GB of RAM to reproduce (Table 3 of Unimocg [ 19 ]). Finally, we exclude one table that only contains non-deterministic results (Table 3 of PPT4J [36]). When trying to reproduce tables, it may become obvious that either the paper or artifact contains some errors, leading to inconsistencies between the two. As part of this work, we found 20 cases where either the paper or the artifact is erroneous, which we discuss in detail in Section 4.6. In these cases, plus three cases where the paper authors claim the results of the artifact could be slightly different from those in the paper [ 40 ], we use the corrected version of the table as the task in our benchmark. Moreover, in some cases, we include only a part of the table as the task: for 15 tables because we could only partially reproduce the results, and for four tables because some results are non-deterministic and hence excluded. 

3.4 Ground Truth Scripts 

For each task, we prepare a ground truth script that reproduces the table from scratch using the artifact. In total, we create 30 full reproduction scripts and 30 last-mile reproduction scripts. Note that we do not utilize these ground truth scripts for our automated judging mechanism (Section 2.5); rather, they serve as executable evidence that reproduction is possible for each table in the evaluation set. To quantitatively characterize the complexity of the ground truth scripts, we report the number of non-blank, non-comment lines of code (LoC) in each script. On average, the non-blank, non-comment LoC is 71.8, with a median of 57. We also manually analyze all the ground truth scripts for how simple the ground truth scripts can be and find that only six scripts are simple, i.e., the expected table can be produced with a single command. Most scripts require multiple commands invocation, complex formatting logic, or sometimes fix to the outdated or buggy artifact code. Even in the case where the ground truth script is simple 2, LLM agents evaluated still make mistakes while formatting the output into a expected table. Table 2 compares Artisan-Bench with existing benchmarks for automated research reproduction. Artisan-Bench is the only benchmark in software engineering and covers several programming languages not represented in the existing benchmarks. As many software engineering experiments involve time-consuming tasks (e.g., dynamic analysis), we set a relatively high time limit of eight hours per task. Although the number of papers covered (23) is lower in Artisan-Bench due to the manual validation required, the number of tasks (60) is comparable to the existing benchmarks. Note that a single task in Artisan-Bench typically corresponds to multiple tasks in the existing benchmarks as a single table usually contains multiple numerical research results.

4 EVALUATION 

To evaluate our approach, we answer the following research questions: 

• RQ1. Effectiveness : How effective is Artisan at generating successful reproduction scripts? 

• RQ2. Efficiency : How efficient is Artisan in terms of monetary cost and execution time? 

• RQ3. Ablation study : What is the impact of our core technical contributions, i.e., output judge, method judge, and format tool? 

• RQ4. Construction and detection of copied-results scripts : How do agents construct copied-results scripts, and how accurate is the method judge at detecting them? 

• RQ5. Discovery of paper-artifact inconsistencies : What paper-artifact inconsistencies does Artisan help discover? 

4.1 Experimental Setup 

4.1.1 Agent Framework and LLM Usage. We use mini-swe-agent [ 42 ], version 1.14.4, as the base agent framework for building Artisan. We implement the preparation mechanisms, judging mecha-nism, and the format tool, in Python 3.12.10. As the LLM behind the format tool and method judge, we use gpt-5-mini-2025-08-07. 

4.1.2 Baselines. As baselines, we select the default versions of mini-swe-agent [ 42 ], SWE-agent [ 48 ], and OpenHands [ 44 ]. We choose mini-swe-agent, which Artisan builds on, because comparing against mini-swe-agent enables a thorough empirical study of Artisan’s contributions. We choose SWE-agent and OpenHands as established state-of-the-art approaches for general software engi-neering tasks. 

4.1.3 Settings. For all baselines and Artisan, we experiment with two proprietary models (gpt-5.1-2025-11-13 and gpt-5-mini-2025-08-07) and one open-weight model (DeepSeek V3.2 Reasoner). We use a step limit of 30 and a cost limit of USD 1 per task. For mini-swe-agent and OpenHands, we provide the identical user prompt instructing on the table to reproduce and workflows to follow. For SWE-agent, we use the same user prompt as in mini-swe-agent but add one instruction to adapt SWE-agent’s hard-coded APR task to our setting. For Artisan, we additionally provide instructions for our preparation mechanisms, judging mechanism, and the format tool.

4.2.3 Reasons for failure. Across the 10 tasks that no approach (including Artisan) can repro-duce, we observe two failure modes. (1) Irreproducible checked-in results (5/10): In Bouzenia and Pradel [ 13 ], prior Jupyter runs for Tables 1–5 are checked in, but re-running the notebook no longer yields them because the underlying code/data that produced results of the paper is missing. This suggests a need for stronger guidance when there are paper–artifact inconsistencies. (2) Buggy artifact code (1/10): In AXA [ 37 ], a Docker-shipped bash script used for counting changes in lines of code contains errors. The erronous bash script contains a stray + and a missing term in a sum, leading to different results. Agents execute the erronous script, obtain incorrect counts, and fail to repair it. This motivates automated program repair technique that is applicable to the artifacts to aid the reproduction process.

4.4 RQ3: Ablation Study 

Table 4 summarizes our ablation study, which uses three variants of Artisan that exclude the output judge, the method judge, and the format tool, respectively. Omitting the output judge increases the number of reproduction scripts with runtime error (4 to 13) and mismatched results (10 to 19). In principle, agents can check for errors or mismatches themselves by running the generated scripts, but enforcing this check on submission attempt makes a significant difference. Reduction in monetary cost and wall-clock time is not a desirable sign; it merely indicates that the agent exits early with non-functional reproduction scripts. Without the method judge, we see a dramatic increase in copied-results scripts (2 to 21), showing that the method judge is crucial for inducing the agent to seek a more complete reproduction method. Omitting the format tool decreases successful reproductions (44 to 35) and increases copied-results scripts (2 to 9). These two effects are related: When given the liberty to format the reproduced table as they like, the agent often directly copies the expected results. Providing the format tool to the agent reduces such behavior by simplifying the task of correctly formatting the output. 

4.5 RQ4. Construction and detection of copied-results scripts 

4.5.1 How do agents construct copied-results scripts? During development and evaluation of Artisan, we observed many cases where agents generated copied-results scripts (Section 2.5.2). While similar misalignment issues are reported in other domains [ 8 , 32 ], to our knowledge, this is the first report of such misalignment in artifact evaluation. To understand how agents construct copied-results scripts, we sample 30 cases and manually classify their strategies: In 12/30 cases, agents directly copy checked-in research results. In 7/30 cases, agents convert the paper PDF to text and copy expected tables. In 7/30 cases, agents copy outputs from included Jupyter notebooks. In 3/30 cases, agents infer results from surrounding paper text despite table obfuscation. Finally, in one case, an agent downloads the preprint by searching for it on arXiv. These diverse strategies indicate that preventing specific misalignment types is infeasible, confirming our decision to use a method judge to detect copied-results scripts.

4.6 RQ5. Discovery of Paper-Artifact In 

By using Artisan, we encounter 20 inconsistencies between the papers or the artifacts. We identify these cases by noticing inconsistencies between the paper results and the output of the reproduction script when the reproduction script appears to contain no apparent error. We do not include cases where artifacts themselves warn about possible inconsistencies; thus, all 20 cases we report are newly discovered inconsistencies that have not been disclosed publicly before. Table 6 summarizes the paper-artifact inconsistencies we find. Overall, we find 20 paper-artifact inconsistencies from 13 papers, i.e. 57% of the studied papers. All these 13 paper underwent an artifact evaluation in major SE conferences and were awarded the Artifacts Evaluated – Reusable badge. Still, with the aid of Artisan, we find inconsistencies in up to 29 entries in a single table, with values differing by up to 8.13 × 10 285960 . The discovery of paper-artifact inconsistencies is supported by Artisan and the reproduction scripts it generates, but it is not entirely automated. Given the reproduction script, it is ultimately up to a human expert to decide if there is indeed a bug, and if so, where the bug lies. To provide more insight into the paper-artifact inconsistencies we find, we manually analyze the reasons for the inconsistencies, leading to five categories: 

• Artifact difference (9/20): The artifact (code and data) used for the paper differs from the artifact publicly available. 

• Transcription error (5/20): Inconsistencies that likely stemmed from the manual process of transcribing artifact values to the paper. 

• Rounding error (4/20): Inconsistencies due to incorrect rounding. 

• Artifact error (2/20): Errors in the artifact. We report our findings (excluding round errors, as they are obvious but minor issues) to the paper authors, and 14/16 of the reported inconsistencies are confirmed. 5 THREATS TO VALIDITY 

Our results are influenced by the step limit (30 steps) and cost limit ($1). With GPT-5.1, step limits are hit by SWE-agent (18/60), OpenHands (22/60), mini-swe-agent (6/60), and Artisan (16/60), suggesting that early stopping in most tasks limits effectiveness more than incomplete exploration. The cost limit represents a reasonable budget for automated reproduction of a single table. Additionally, our evaluation is limited to Docker-packaged papers without non-public APIs or GPUs, as described in Section 3.2. We focus on this subset to reduce manual and computational costs. 

6 RELATED WORK 

Artifact Evaluation. Artifact evaluation was introduced at ESEC/FSE 2011 [ 26 ] and adopted by the programming languages [ 27 ], security [ 35 ], and database communities [ 7]. Recent work has examined artifact evaluation processes, identifying reproducibility and reusability as primary goals [ 21 ] and showing that evaluated artifacts are more available, documented, and maintained [ 46 ]. Others measured reproducibility of LLM-based software engineering research, reporting that half of the papers with the Artifacts Evaluated – Reusable badge were incomplete or non-functional one year later [ 5 ]. However, none of these efforts automate artifact evaluation itself. Our work tackles this problem and paves the way for automated artifact evaluation research. 

Automated Research Reproduction. Recent work leverages LLM agents for reproducibility, defined as enabling a different team to obtain similar experimental outcomes using the same setup [ 6 ].  

> , Vol. 1, No. 1, Article . Publication date: February 2026. 18 Doehyun Baek and Michael Pradel

SUPER [ 10 ] introduces a benchmark evaluating agents’ ability to set up and execute tasks from research repositories. CORE-Bench [ 39 ] also evaluates agents on reproducing research results, expanding to more scientific disciplines (e.g., social science, medicine), while REPRO-Bench [ 24 ]targets social science papers. Unlike SUPER and CORE-Bench, REPRO-Bench is rating-based: the agent outputs a reproducibility score from 1 to 4. Heye et al. [ 22 ] develop an LLM-based system that supports artifact evaluation by outputting a reproducibility score, setting up environments, and identifying pitfalls. Our approach and benchmark differ from prior work as follows: (1) We formulate the problem as a code generation task, enabling independent assessment of generated reproduction scripts. (2) We provide an automated judging mechanism that evaluates both the output and the method of reproduction. (3) We cover diverse software engineering techniques and programming languages not previously addressed. 

Automated Research Replication. There is a related but distinct problem of research replication, defined as the ability of a different team to obtain similar experimental outcomes using a different setup [ 6]. Liang et al. [ 31 ] studied LLMs for replicating empirical software engineering research, finding them to effective at identifying assumptions and boilerplate code. PaperBench [ 41 ] evaluates LLM agents on replicating ML papers and reports a top replication score of 21.0%, below the human baseline. PaperCoder [ 38 ] is a multi-agent LLM framework for automated research replication via planning, analysis, and generation. Xiang et al. [ 47 ] introduce SciReplicate-Bench, a replication benchmark for 36 NLP papers an multi-agent approach for it. Our work fundamentally differs in the problem addressed (reproduction vs. replication) and in augmenting the real-world artifact evaluation process. 

General Software Engineering LLM Agents. There is a growing trend of using LLM agents for software engineering tasks. RepairAgent [ 11 ] is the first LLM agent for automated program repair. ExecutionAgent [ 14 ] automatically sets up projects and runs test suites. SWE-agent [ 48 ] enhances LLM agents’ ability to use computers for SWE-Bench [ 25 ]. MLE-Bench [ 15 ] benchmarks LLM agents in ML engineering. Our work differs by focusing on automated artifact evaluation. 

7 CONCLUSION 

We present Artisan, the first agent that automates the process of artifact evaluation with LLM agents. Our main technical contribution is the problem definition of reproducing research results as a code generation task and development of the automated judging mechanism. To evaluate Artisan, we develop Artisan-Bench, a new benchmark that evaluates LLM agents’ ability to reproduce tables from software engineering research papers. Our evaluation shows that Artisan is effective and efficient. Furthermore, Artisan has added benefit of aiding the discovery of paper-artifact inconsistencies. We envision Artisan paving the way for further research in the task of automated artifact evaluation, empowering researchers to automatically reproduce previous research and furthering the advancement of science.
