## 摘要
诸如 Scratch 之类的块状编程环境在低代码教育中发挥着核心作用，然而，评估 AI 智能体通过图形用户界面 (GUI) 构建程序的能力仍未得到充分探索。我们推出了 ScratchWorld，这是一个用于评估多模态 GUI 智能体在 Scratch 中执行“通过构建进行编程”任务的基准测试。基于“使用-修改-创造” (Use-Modify-Create) 教学框架，ScratchWorld 包含 83 个精心挑选的任务，涵盖四个不同的问题类别：创建、调试、扩展和计算。为了严格诊断智能体失败的原因，该基准测试采用了两种互补的交互模式：原始模式需要细粒度的拖放操作以直接评估视觉运动控制，而复合模式则使用高级语义 API 将程序推理与 GUI 执行解耦。为确保评估的可靠性，我们提出了一种基于执行的评估协议，通过浏览器环境内的运行时测试来验证所构建 Scratch 程序的逻辑正确性。对最先进的多模态语言模型和 GUI 智能体进行的广泛实验揭示了显著的“推理-行动”差距，强调了尽管具备强大的规划能力，但在细粒度 GUI 操作方面仍面临持续挑战。

## Abstract
Block-based programming environments such as Scratch play a central role in low-code education, yet evaluating the capabilities of AI agents to construct programs through Graphical User Interfaces (GUIs) remains underexplored. We introduce ScratchWorld, a benchmark for evaluating multimodal GUI agents on program-by-construction tasks in Scratch. Grounded in the Use-Modify-Create pedagogical framework, ScratchWorld comprises 83 curated tasks spanning four distinct problem categories: Create, Debug, Extend, and Compute. To rigorously diagnose the source of agent failures, the benchmark employs two complementary interaction modes: primitive mode requires fine-grained drag-and-drop manipulation to directly assess visuomotor control, while composite mode uses high-level semantic APIs to disentangle program reasoning from GUI execution. To ensure reliable assessment, we propose an execution-based evaluation protocol that validates the functional correctness of the constructed Scratch programs through runtime tests within the browser environment. Extensive experiments across state-of-the-art multimodal language models and GUI agents reveal a substantial reasoning--acting gap, highlighting persistent challenges in fine-grained GUI manipulation despite strong planning capabilities.

*   **研究动机**：Scratch 是低代码教育的核心工具，但初学者常在程序逻辑理解和 GUI 操作（如拖拽积木）上遇到困难。虽然现有的 AI 助手能提供文字建议，但无法直接在界面上演示操作。
*   **核心挑战**：现有的 GUI 智能体基准（如针对网页或移动端）多侧重于导航和信息检索，而 Scratch 要求智能体通过长程、组合式的拖拽操作来“构建程序”（Program-by-construction）。这不仅需要极高的视觉定位精度，还需要对程序逻辑有深刻的结构化理解。

### 2. 论文提出的方法论
论文引入了 **ScratchWorld**，这是首个专门用于评估 GUI 智能体在块编程环境中构建程序能力的基准。
*   **任务框架**：基于“使用-修改-创造”（Use-Modify-Create）教学框架，设计了 83 个任务，分为四类：**Create（创建）**、**Debug（调试）**、**Extend（扩展）**和**Compute（计算）**。
*   **双模式交互协议**：
    *   **原始模式 (Primitive Mode)**：提供屏幕截图和 UI 元素列表，要求智能体执行原子操作（点击、拖拽、输入）。旨在测试智能体的**视觉运动控制**。
    *   **复合模式 (Composite Mode)**：提供程序伪代码，通过高级语义 API（如 `add_block`, `connect_blocks`）进行交互。旨在**隔离程序推理能力**，排除 GUI 操作干扰。
*   **基于执行的评估协议**：利用 Playwright 驱动浏览器，通过 Scratch VM 实时监控程序内部状态（如坐标、变量值），利用 JavaScript 编写的单元测试来验证功能正确性，而非仅对比视觉相似度。

*   **数据集/场景**：83 个精心设计的 Scratch 任务，平均每个任务涉及 9.14 个积木修改，最复杂的涉及 32 个。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了 10 余种主流模型和框架在两种模式下的表现，并针对失败原因进行了多维度的消融和诊断实验。
*   **充分性与客观性**：
    *   **解耦分析**：通过双模式设计，成功将“想不出逻辑”和“手笨操作不了”这两个失败维度分开，实验逻辑严密。
    *   **诊断深入**：不仅给出了成功率，还通过单步实验量化了像素级的空间误差（Spatial Error），并利用感知 QA 排除感知能力的干扰。

### 6. 论文的主要结论与发现
*   **推理-行动差距 (Reasoning–Acting Gap)**：这是最核心的发现。顶尖模型（如 Claude-4.5）在复合模式下成功率可达 **78.31%**，但在原始模式下骤降至 **14.46%**。这表明模型“脑子够用”，但“手跟不上”。
*   **终点定位是瓶颈**：单步实验显示，即使告知起始点，模型在预测“积木该往哪儿放”时的准确率依然极低（23%-32%）。
*   **感知并非主因**：模型在静态感知 QA 中表现优异（GPT-5 达 90.5%），说明它们能看懂界面，但在动态操作时的闭环执行精度不足。
*   **现有框架失效**：在其他基准中表现良好的智能体框架（如 AWM）在 Scratch 环境中表现平平，主因是拖拽操作的高失败率导致其无法积累成功的轨迹。

### 7. 优点
*   **填补空白**：首次针对低代码/块编程环境提出了系统性的 GUI 智能体评估基准。
*   **评估严谨**：采用基于运行时的功能验证（单元测试），比传统的视觉匹配或轨迹匹配更具权威性。
*   **诊断工具完备**：提供的单步拖拽和感知 QA 基准为后续研究指明了改进方向（即提升高精度空间定位）。

### 8. 不足与局限
*   **任务规模有限**：83 个任务虽然复杂度高，但样本量相对较小，可能难以完全覆盖 Scratch 的所有长尾功能。
*   **平台局限性**：目前仅针对 Scratch 平台，对于其他低代码平台（如 Blockly 或工业级低代码工具）的泛化性有待验证。
*   **反馈机制单一**：实验中智能体主要依赖视觉反馈，缺乏像人类那样在拖拽过程中的实时“吸附感”或触觉反馈模拟。
