## 摘要
基于大语言模型（LLM）的智能体在自动化软件工程任务（如静态漏洞修复）中展现出强大的能力，这已在 SWE-bench 等基准测试中得到证实。然而，在现实世界中，成熟软件的开发通常取决于复杂的需求变更和长期的功能迭代——这一过程是静态、单次修复范式无法捕捉的。为了弥补这一差距，我们提出了 SWE-CI，这是首个基于持续集成（CI）循环构建的代码库级基准测试，旨在将代码生成的评估范式从静态、短期的“功能正确性”转向动态、长期的“可维护性”。该基准测试包含 100 个任务，每个任务平均对应于真实代码库中跨越 233 天和 71 次连续提交的演进历史。SWE-CI 要求智能体通过数十轮的分析和编码迭代来系统地解决这些任务。SWE-CI 为评估智能体在长期演进过程中维持代码质量的能力提供了宝贵的见解。

## Abstract
Large language model (LLM)-powered agents have demonstrated strong capabilities in automating software engineering tasks such as static bug fixing, as evidenced by benchmarks like SWE-bench. However, in the real world, the development of mature software is typically predicated on complex requirement changes and long-term feature iterations -- a process that static, one-shot repair paradigms fail to capture. To bridge this gap, we propose \textbf{SWE-CI}, the first repository-level benchmark built upon the Continuous Integration loop, aiming to shift the evaluation paradigm for code generation from static, short-term \textit{functional correctness} toward dynamic, long-term \textit{maintainability}. The benchmark comprises 100 tasks, each corresponding on average to an evolution history spanning 233 days and 71 consecutive commits in a real-world code repository. SWE-CI requires agents to systematically resolve these tasks through dozens of rounds of analysis and coding iterations. SWE-CI provides valuable insights into how well agents can sustain code quality throughout long-term evolution.

*   **研究背景**：在真实软件工程中，代码是随时间演进的。一个“硬编码”的补丁可能通过当前测试，但会积累技术债，导致后续功能迭代困难。论文旨在填补这一空白，将评估视角从“一次性修复”转向“持续集成（CI）下的长期维护”。

### 2. 论文提出的方法论
*   **核心思想**：利用真实代码库的历史演进过程，要求智能体从一个历史提交点（Base Commit）出发，通过多轮迭代，最终达到数月后的目标状态（Oracle Commit）。
*   **任务形式化**：
    *   定义需求生成函数 $require_T$ 和代码修改函数 $code_T$。
    *   不同于静态评估，SWE-CI 采用迭代循环：$c_{i+1} = code_T(c_i, require_T(c_i, c^*))$，使前期决策的后果能传播到后续迭代中。
*   **核心指标 (EvoScore)**：
    *   **归一化变化量 $a(c)$**：衡量当前代码相对于初始状态的进步或退步。
    *   **EvoScore ($e$)**：采用未来加权平均值公式 $e = \frac{\sum_{i=1}^N \gamma^i a(c_i)}{\sum_{i=1}^N \gamma^i}$。其中 $\gamma \ge 1$。当 $\gamma$ 较大时，指标更看重后期演进的稳定性，从而惩罚那些虽然早期得分高但因设计差导致后期无法维护的行为。
*   **双智能体协议 (Architect-Programmer)**：
    *   **架构师智能体 (Architect)**：分析测试失败原因，定位根因，并编写高层级、增量式的需求文档（XML 格式）。
    *   **程序员智能体 (Programmer)**：根据需求文档理解意图、制定计划并实施代码修改。

### 6. 论文的主要结论与发现
*   **能力演进**：LLM 的代码维护能力正在加速提升，2026 年后发布的模型在处理长期演进任务上显著优于早期模型。

### 7. 优点
*   **范式创新**：首次将代码评估从静态快照提升到动态 CI 循环，更贴近真实软件开发。
*   **数据质量高**：基于真实世界长达半年的演进历史，而非人工构造的简单任务。
*   **评估维度深**：通过 EvoScore 巧妙地量化了“技术债”对后续开发的影响。

### 8. 不足与局限
*   **样本规模限制**：100 个任务虽然质量高，但相比 SWE-bench 的规模仍较小，可能存在一定的统计偏差。
*   **框架依赖**：评估结果高度依赖于预设的“架构师-程序员”双智能体框架和 Prompt 设计，不同的 Agent 架构可能会影响模型能力的发挥。
*   **语言局限性**：目前仅针对 Python 语言，尚未覆盖 Java、C++ 等其他大型企业级应用常用的语言。
*   **成本高昂**：多轮迭代导致评估单个模型的 Token 成本极高，不利于社区大规模频繁测试。
