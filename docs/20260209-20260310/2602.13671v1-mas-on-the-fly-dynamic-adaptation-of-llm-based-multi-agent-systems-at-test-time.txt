Abstract 

Large Language Model (LLM)-based multi-agent systems (MAS) have emerged as a promising paradigm for solving complex tasks. However, existing works often rely on man-ual designs or “one-size-fits-all” automation, lacking dynamic adaptability after deployment. Inspired by how biological systems adapt, we introduce MASFly, a novel multi-agent frame-work enabling dynamic adaptation at test time. To adapt system generation, MASFly employs a retrieval-augmented SOP instantiation mech-anism that leverages a self-constructed repos-itory of successful collaboration patterns, en-abling the LLM to assemble customized MASs for new queries. For adaptive execution, MAS-Fly incorporates an experience-guided super-vision mechanism, where a dedicated Watcher agent monitors system behaviors with reference to a personalized experience pool and provides real-time interventions. Extensive experiments demonstrate that MASFly achieves state-of-the-art performance, most notably a 61.7% success rate on the TravelPlanner benchmark, while ex-hibiting strong task adaptability and robustness. 

1 Introduction 

The remarkable advancements in Large Language Models (LLMs) have transformed the field of ar-tificial intelligence, showing remarkable abilities in reasoning and generation (Achiam et al., 2023; Liu et al., 2024; Yang et al., 2025). However, tackling complex, multi-faceted problems often ex-ceeds the capacity of a single LLM (Talebirad and Nadiri, 2023). This has led to the rapid emergence of LLM-based multi-agent systems (MAS) (Hong et al., 2023; Wu et al., 2024), where multiple agents take on specialized roles and work collaboratively, demonstrating human-like collective intelligence for advanced problem-solving. To address this, we draw inspiration from bio-logical adaptation: organisms utilize a conserved genetic toolkit encoding evolutionary experience to express diverse traits suited to varying environ-ments ( phenotypic plasticity ), and leverage regu-latory systems for real-time adjustment to main-tain stability amidst internal and external pertur-bations ( homeostatic regulation ), as illustrated in Figure 1. Inspired by this, we propose MASFly, a novel multi-agent framework that achieves on-the-fly adaptation without extra training, covering both adaptive system generation and execution. Specifically, to navigate the combinatorially large design space of MAS, we mirror the logic of plasticity through a retrieval-augmented SOP instantiation mechanism for dynamic system gen-eration. MASFly maintains an SOP repository that stores successful collaboration patterns across tasks, serving as the “genetic units” of the system. Given a new query, a retrieval-augmented gener-ation (RAG) process selects relevant patterns and adaptively expresses them, instantiating tailored agent roles and communication structures. Further-more, to handle the non-stationary multi-agent envi-ronment—where agent actions reshape the context and external shifts occur unexpectedly—we incor-porate a mechanism of homeostatic regulation via experience-guided process supervision. A global Watcher agent acts as a regulator, monitoring the system while referring to a personalized experience pool as guidance, enabling on-the-fly adjustments (e.g., replacing malfunctioning agents) and forming a resilient closed-loop execution process. Extensive experiments demonstrate that MASFly achieves state-of-the-art performance, most notably a 61.7% success rate on the TravelPlanner benchmark. The contributions are summarized as follows: • We propose MASFly, a novel multi-agent frame-work enabling dynamic adaptation at test time without extra training, achieving both adaptive system generation and execution. • We design a retrieval-augmented SOP instantia-tion mechanism for dynamic system generation, which leverages an SOP repository of successful collaboration patterns to construct tailored MAS. • We design an experience-guided process super-vision mechanism, which integrates a global Watcher agent and a personalized experience pool for real-time adaptation during execution. • Extensive experiments across diverse tasks demonstrate that MASFly not only outperforms strong baselines, but also exhibits superior task adaptability and robustness. 

2 Related Work 

2.1 LLM-based Multi-Agent System 

The evolution of LLM-based MAS has undergone a transition from manually designed frameworks to increasingly automated and adaptive construc-tion. Early systems, such as MetaGPT (Hong et al., 2023) and ChatDev (Qian et al., 2023), rely heav-ily on handcrafted SOPs to configure agents roles and workflows. Subsequent research introduces semi-automated mechanisms. For example, Agent-Verse (Chen et al., 2023), EvoAgent (Yuan et al., 2024), and MegaAgent (Wang et al., 2025) lever-age LLMs to automatically generate diverse agent roles. Other studies such as GPTSwarm (Zhuge et al., 2024) and G-Designer (Zhang et al., 2024a), which employ external models to learn the commu-nication topology across agents. Recently, the focus has shifted toward fully au-tomated MAS generation driven by LLMs. These methods typically prompt the LLM to generate executable code that represents the entire sys-tem. Search-based methods (e.g., ADAS (Hu et al., 2024), AFlow (Zhang et al., 2024b), AgentSquare (Shang et al., 2024), SwarmAgen-tic (Zhang et al., 2025c) use LLMs to identify an "optimal" system configuration within a pre-defined search space. However, these methods rely on fixed system, lacking dynamic adaptabil-ity to diverse queries. Training-based approaches ,such as MAS-GPT (Ye et al., 2025) and FlowRea-soner (Gao et al., 2025), train a meta-agent via su-pervised fine-tuning (SFT) or reinforcement learn-ing (RL) to autonomously generate MAS configu-rations. While more adaptive to different queries, these approaches need extensive data and struggle to generalize. Moreover, most existing works ne-glect execution-time adaptability, leading to brittle open-loop pipelines. 

2.2 Learning at Test Time for LLMs 

Learning at test-time refers to techniques that en-able LLMs to adapt or improve their behavior dur-ing inference, without updating model parameters. These methods allow an LLM to leverage external knowledge, adjust its reasoning strategy, or correct its own outputs on the fly. Retrieval-augmented genera-tion (RAG) (Lewis et al., 2020) leverages relevant documents or memories retrieved at test time to fill knowledge gaps and guide generation. Test-time scaling methods (Snell et al., 2024) expand LLM’s inference-time computation—such as sam-pling or reasoning path exploration—to improve accuracy for individual instances. In addition, test-time adaptation techniques (Wang et al., 2020) aim at adapting a pre-trained model from the source do-main to unlabeled data in the target domain. 2.3 Memory in Multi-Agent System 

Memory plays a vital role in enabling LLM agents to accumulate experiences and enhance reason-ing capabilities (Wang et al., 2024a). Research on single-agent memory has advanced consider-ably, evolving from simple RAG-style context ex-tensions (Packer et al., 2023) to cognitively in-spired architectures that support cross-trial learn-ing (Xu et al., 2025; Wang et al., 2024b). How-ever, memory mechanisms tailored for MAS re-main largely underexplored (Tang et al., 2025). Re-cent efforts (Zhang et al., 2025a) have attempted to manage MAS interaction histories and experiences via hierarchical graph structures, offering external memory banks for different MASs. 

3 Proposed Method 

3.1 Overall Framework 

The design of MASFly is motivated by the need for MAS to move from static, one-size-fits-all con-figurations to adaptive, reliable, and continuously improving collaboration at test time. To achieve this, MASFly draws inspiration from biological adaptation principles of phenotypic plasticity (i.e., the ability to express different traits based on envi-ronmental cues), and homeostatic regulation (the real-time adjustment capacity to maintain stability). Inspired by these principles, MASFly operates through a closed-loop, three-stage process, as illus-trated in Figure 2: (1) SOP-Guided System Gener-ation , (2) Process Supervised Execution , and (3) 

Reflective Experience Distillation . MASFly begins with dynamic system generation via a retrieval-augmented SOP instantiation mechanism. Treat-ing the SOP repository as a conserved “genetic toolkit” containing successful collaboration pat-terns, MASFly employs a hybrid retrieval strategy to identify query-related patterns. An SOP Agent then adaptively “expresses” these retrieved units, instantiating a query-specific MAS. During execu-tion, MASFly applies an experience-guided pro-cess supervision mechanism. Acting as a regulator, a global Watcher agent continuously monitors the system with the reference to a personalized expe-rience pool , and dynamically adjusts agents when abnormal behaviors are detected. After execution, the LLM reflects on the outcome. It then either summarizes effective SOP to update the reposi-tory, or refines the system and records lessons from failures for future execution. Through this design, MASFly not only adapts its structure to immediate tasks but also continuously enhances its resilience and capability for future collaborations. 33.2 SOP-Guided System Generation 

Constructing adaptive and high-quality MAS solely from an LLM’s internal knowledge is challeng-ing, as the design space is vast and models of-ten lack task-specific priors (Ye et al., 2025). In nature, biological development navigates envi-ronmental complexity using a conserved genetic toolkit—encoding evolutionarily proven patterns while allowing for flexible expression across dif-ferent contexts. Mirroring this strategy, MASFly introduces a retrieval-augmented SOP instantiation mechanism, which leverages an SOP Repository 

that systematically records successful collabora-tion patterns, to guide the LLM in adpative system generation through RAG.

Each SOP in the repository is bound to a concrete case, enabling the system to retrieve suitable collaboration templates when facing new queries with analogous require-ments. Hybrid Retrieval and Instantiation 

Given a new query, MASFly aims to identify rel-evant collaboration patterns from SOP repository, leveraging past experience to generate a tailored MAS. Specifically, for the query Q, we first prompt an LLM to identify the need N , and then compute a hybrid retrieval score based on a weighted com-bination of external query similarity and internal need similarity to retrieve similar cases Ci :

s(Q, C i) = λ · Sim (Q, Q i)+ (1 − λ) · Sim (N, N i), (1) where λ ∈ [0 , 1] is a hyperparameter that bal-ances the contribution of the two components, and Sim (·, ·) denotes the cosine similarity based on an embedding model. The SOPs in the top-K most similar cases Ci are retrieved from the repository, guiding an LLM to dynamically generate an operation procedure (OP) that fits the current query: 

OP = LLM (Q, N, {Si}Ki=1 ), (2) where OP , structured similarly to an SOP, specifies the query-specific agent prompts and communica-tion workflow. Consequently, MASFly transforms reusable procedural templates into concrete MASs, preserving effective coordination patterns while flexibly adapting to novel task requirements.

Personalized Experience Pool 

Effective supervision requires the ability to iden-tify and handle typical anomalies. To this end, we construct a Personalized Experience Pool (PEP) through a reflection mechanism by an LLM (de-tailed in Section 3.4), which stores common er-ror patterns and their corresponding remedies, sup-porting the supervision module continuously refine agents behaviors. As shown in Figure 4, the each record is formalized as < Q i, F i, P i >, which com-prises the User Query Qi, the Failure Cause Fi

(e.g., task misallocation, tool misuse), and the set of Agent-Wise Experiences Pi = {p1, p 2, . . . , p m},where each pk encapsulates an Error Attribution 

diagnosing the agent’s specific contribution to the failure and an Improvement Strategy proposing cor-rective insights to prevent future recurrence. By or-ganizing agent-level failure cases and improvement strategies from past executions, the PEP serves as a knowledge base for adaptive monitoring and real-time behavioral refinement. 

Process Supervision and Optimization 

During system execution, the supervision is carried out by a global Watcher Agent, which oversees system behavior, referencing both the OP and PEP. By retrieving relevant failure cases (indexed by the same queries as the SOP repository), the Watcher gains priors about potential pitfalls and uses them to implement dynamic, experience-informed inter-ventions. Specifically, the Watcher conducts dual-level supervision: • Inter-Agent Communication : verifying whether agents follow the prescribed workflow in the OP, maintain output consistency, and avoid unproduc-tive conversational loops; • Agent–Environment Interaction : assessing whether agents properly invoke designated tools, respond appropriately to dynamic contexts, and prevent repetitive or irrelevant outputs. When an anomaly is detected, the Watcher can take either of two possible actions for optimization: • Experience-Augmented Guidance : sending a feedback message to the faulty agent, incorporat-ing insights from the PEP to enhance the agent’s reasoning or operational strategy in real time. • Dynamic Agent Replacement : removing the criti-cally flawed agent and dynamically instantiates a new one Ai to resume the task, ensuring work-flow continuity. To balance controlibity and efficiency, MAS-Fly supports flexible intervention policies, such as triggering supervision every M communication rounds, every L environment-interaction steps, or capping the total number of interventions. Re-lated experiments are provided in Appendix C.1. Through this experience-guided supervision mech-anism, MASFly enhances its ability to detect and address execution deviations, enabling dynamic adaptation throughout the entire execution process. 

3.4 Reflective Experience Distillation 

Upon task completion, MASFly conducts a reflec-tion process guided by evaluation outcomes (train-ing set only). For successful executions, the LLM distills the generated OP into a generalized and reusable SOP, which is stored in the SOP reposi-tory. For failed executions, MASFly prompts the LLM to diagnose shortcomings in the OP and at-tributing errors to specific agents. These reflective insights serve two purposes: iteratively refining the instantiated SOP for successful task completion, and enriching the PEP to form agent-level priors that guide the process supervision during future executions. 4 Experiment 

4.1 Experimental Setup Benchmarks. We evaluate MASFly on five benchmarks involving tool calls covering three domains: (1)long-horizon planning: TravelPlan-ner (Xie et al., 2024),which provides two evalua-tion modes: Two-Stage and Sole-Planning ; (2)gen-eral question answering: GAIA (Mialon et al., 2023), consisting of questions categorized into three difficulty levels; and (3) code generation: Hu-manEval (Chen et al., 2021), MBPP (Austin et al., 2021) and MBPP Pro (Yu et al., 2024). The dataset statistics are in Appendix B.3. 

Baselines. We compare MASFly with three series of baselines without additional training: (1) vanilla single LLMs, including CoT (Wei et al., 2022), ReAct (Yao et al., 2022), Reflex-ion (Shinn et al., 2023); (2) hand-craft MAS, in-cluding MetaGPT (Hong et al., 2023); (3) (par-tially or fully) autonomous MASs, including AgentVerse (Chen et al., 2023), EvoAgent (Yuan et al., 2024), GPTSwarm (Zhuge et al., 2024), AFlow (Zhang et al., 2024b), AgentSquare (Shang et al., 2024), MegaAgent (Wang et al., 2025) and SwarmAgentic (Zhang et al., 2025c). 

Implementation details. We utilize Qwen3-235B-A22B-Thinking-2507 (Yang et al., 2025) as LLM backbone via API call, with the temperature set to 0.6. We typically set the number of retrieved SOP K = 2 , and the hyperparameter λ = 0 .3.

To assess the robustness of MAS under adversar-ial conditions, we follow prior work (Zhuge et al., 2024) and introduce a prompt attack during test time on MBPP Pro benchmark, where we force the coding agent to output an empty function. Fig-ure 6 reports the performance before and after at-tack across representative systems. Most methods experience a substantial degradation after attack, indicating their inability to mitigate abnormal agent behaviors once execution begins. In contrast, MAS-Fly maintains almost the same performance. This robustness stems from MASFly’s process supervi-sion: the Watcher identifies the compromised agent, removes it and dynamically instantiates a replace-ment agent to continue execution. These results highlight that MASFly uniquely enables dynamic adaptation at test time, preserve high task accuracy under adversarial or abnormal conditions.

To further illustrate how MASFly dynamically adapts to query requirements and ensures reli-able execution, we present several representative cases in Figure 7. 5 Conclusion 

In this paper, we present MASFly, a novel LLM-based multi-agent framework that achieves dy-namic adaptation at test time without additional training. Drawing inspiration from biological adap-8tation principles, MASFly dynamically constructs query-dependent MAS via retrieval-augmented SOP instantiation. For adaptive execution, it incor-porates an experience-guided process supervision mechanism. Extensive experiments demonstrate that MASFly achieves state-of-the-art performance, exhibiting high adaptability and robustness. 

Limitations 

One limitation of MASFly is that the triggering mechanism of the process supervision relies on pre-defined heuristic rules (e.g., setting the intervention interval to half the number of agents) to initiate in-terventions. This approach may not be optimal for all scenarios. In future work, it is worth explor-ing automated anomaly detection mechanisms that allow the system to autonomously determine the precise timing for intervention, thereby reducing the reliance on manual heuristics.

10 A Comparison with Related Works 

In Table 1, we compare MASFly with representa-tive MAS from multiple perspectives. Here, dy-namic adaptation at test time covers adaptive sys-tem generation and execution without requiring ad-ditional LLM training. Adaptive system generation entails tailoring agent prompts and communication topologies to each input query, while adaptive exe-cution involves dynamically adjusting system con-figurations throughout the execution phase. Cross-task memory refers to the system’s capability to reuse memories or experiences from previous tasks or queries to facilitate the current task. Specifically: • MetaGPT (Hong et al., 2023) is a pioneering framework for constructing LLM-based MAS with diverse roles. However, it relies on fixed, handcrafted agent prompts and communica-tion structures, limiting adaptability to different queries. In addition, it lacks process supervi-sion and experience reuse mechanisms. Similar works include ChatDev (Qian et al., 2023) and AutoGen (Wu et al., 2024). • AgentVerse(Chen et al., 2023) leverages an LLM to automatically generate diverse agent roles based on user queries, enhancing prompt adapt-ability. However, it still employs a static commu-nication structure, making it not a fully adaptive system. It can dynamically replace team mem-bers by self-assessing execution results, but it lacks experience augmentation mechanism. Sim-ilar works include EvoAgent(Yuan et al., 2024). • MegaAgent (Wang et al., 2025) prompts a Boss agent to customize agent roles and does not pre-define communication structures, relying instead on agents to coordinate themselves during ex-ecution. However, this approach may lead to inefficiency and confusion in long-horizon tasks without explicit communication design. It in-troduces a hierarchical monitoring mechanism, where internal validation occur between agents, rather than having an external, global Watcher like MASFly to provide unified process super-vision, including agent communication and en-vironment interaction monitoring. Moreover, it lacks mechanisms for experience reuse. • GPTSwarm (Zhuge et al., 2024) focuses on opti-mizing communication structures among agents through adaptive communication graphs. How-ever, it does not adapt agent prompts for different queries. It also cannot adapt to or handle un-expected situations during testing, and lacks a mechanism for reusing past experience. Similar works include G-Designer (Zhang et al., 2024a). • AgentSquare (Shang et al., 2024) does not in-volve multi-role playing. Instead, it uses an LLM to search for an optimal agentic system on a train-ing dataset, including modules for planning, tool usage and memory. However, it adopts a fixed lin-ear sequential collaboration structure. Addition-ally, it lacks dynamic adaptability after deploy-ment, and lacks experience reuse mechanisms. Similar works include ADAS (Hu et al., 2024), AFlow (Zhang et al., 2024b), MaAS (Zhang et al., 2025b), and SwarmAgentic (Zhang et al., 2025c). Beyond these methods, there are also recent training-based approaches, such as MAS-GPT(Ye et al., 2025) and FlowReasoner(Gao et al., 2025), which train a meta-agent using supervised fine-tuning (SFT) or reinforcement learning (RL) to adaptively generate MAS based on different queries. However, these methods require large amounts of training data and computational re-sources, and they also lack execution adaptabil-ity after deployment and experience reuse mecha-nisms. In contrast, MASFly is a training-free frame-work that uniquely enables adaptive generation of entire query-dependent systems. This is achieved through the retrieval-augmented SOP instantiation mechanism, where both agent prompts and commu-nication topology are dynamically customized. It also supports adaptive execution by incorporating a process supervision and optimization mechanism. Moreover, MASFly features an inherent memory mechanism that enables experience augmentation by reusing successful system-level experience from the SOP repository and agent-level experience of failure from the PEP, significantly enhancing the system’s capacity for continuous improvement. 

B Implementation Details 

B.1 Details of the Repository B.1.1 Repository Initialization 

To initialize the SOP repository, we randomly sam-ple a subset of training instances and construct an initial collection of SOP cases. For the Trav-elPlanner and GAIA benchmarks, where multiple tools are available, we encourage the LLM (Qwen3-235B) to generate diverse agent roles, each respon-11 sible for distinct tools, together with a dedicated final-answer submitter to ensure response qual-ity. For coding benchmarks such as HumanEval, queries are often simple and the Bash tool is unnec-essary. To maintain system efficiency, we prompt the LLM to begin by designing a minimal MAS (e.g., a single-agent team). If the resulting solu-tion is incorrect, the LLM is then guided to in-crementally introduce additional roles to enhance execution quality. This staged construction strategy yields an initial repository that is both diverse and computationally efficient. 

B.1.2 Repository Update 

Updates to both the SOP repository and the Per-sonalized Experience Pool (PEP) are carried out during the third stage of our method, reflective ex-perience distillation. Note that this process takes place only on labeled training data , not on the test set. During testing, ground truth is not visible to the system; thus, no iterative reflective correction is performed. For successful executions, the LLM is prompted to summarize the current task-specific OP into a more abstract SOP. This involves converting agent instructions that rely heavily on the user’s query into generalized, reusable directives. During this process, the LLM also references previously re-trieved SOPs, extracting and reusing valuable in-structions. The resulting SOP template is then stored in the SOP repository. For failed executions, the LLM receives the de-tailed evaluation outcome together with the full MAS execution trajectory (i.e., communication logs and tool-use traces). It then performs reflec-tion to diagnose the failure, attribute individual agent errors, and update the personalized experi-ence pool accordingly. The reflective feedback is also used to revise the task’s OP—such as ad-justing agent roles, modifying the communication structure, or refining agent instructions and tool usage—and the task may be re-executed until suc-cess or until the maximum number of iterations is reached. Because SOPs are compact, high-level, and query-agnostic, only a small amount of training data is needed to build the SOP repository and the corresponding PEP. In practice, the final SOP repos-itory contains fewer than 100 entries across all tasks (see Table 5). Therefore, no additional deletion or curation strategy is required. If future applications involve significantly more diverse or complex sce-narios, lightweight scalability can be achieved by clustering SOP embeddings and retaining cluster centroids to maintain a compact repository. 

B.2 Agent Communication Details 

To transcend the constraints of static blueprints and enhance adaptability, MASFly adopts a bidi-rectional communication mechanism: each agent can send messages to its downstream agents as pre-scribed by OP , but can also communicate upward to the agents from which it previously received messages—for example, requesting clarification or more complete information. The agent’s output format adheres to the Re-Act (Yao et al., 2022) structure, meaning it first outputs a thought, followed by an action, which may involve either using a tool or sending a mes-sage to another agent. All inter-agent messages are stored in a global message pool, which enables efficient and parallel processing of communication events, ensuring both scalability and consistency during execution. All tool invocations by agents are logged into tool usage history, allowing Watcher to review. When the Watcher decides to remove an anomalous agent, all related entries of that agent in the message pool and the tool usage history are completely purged, thereby thoroughly eliminating the impact of the anomalous agent. 

B.3 Experimental Setup B.3.1 Datasets and Evaluation Metrics 

Following Zhuge et al. (2024); Wang et al. (2025), we evaluate MASFly and compared methods on five representative benchmarks involving tool calls across three domains: • Long-horizon planning: TravelPlanner (Xie et al., 2024) includes two modes: two-stage and 

sole-planning . We mainly focus on the two-stage mode, where the model is required to use six tools to gather information from a sandbox en-vironment and formulate a complete travel plan. Since most previous methods have not adopted this more challenging setting but rather the sole-planning mode, where all necessary information is provided without tool use, we also evaluate our method under this easier mode for complete-ness. Following the original evaluation setting, the evaluation metrics include delivery rate, com-monsense constraint pass rate, hard constraint pass rate, and final plan pass rate. 12 • General QA: GAIA (Mialon et al., 2023) is a benchmark specifically designed to test the gen-erality of AI assistants on real-world, knowledge-intensive questions. Answering GAIA questions typically requires a combination of reasoning, web searching, and tool utilization. Follow-ing (Zhuge et al., 2024), all methods are provided only with the Google Search tool via SerpAPI 1

and a local file processing tool, without access to any paid browsing APIs or multimodal tools. Accuracy is used as the evaluation metric. • Coding: (1) HumanEval (Chen et al., 2021) is a hand-crafted benchmark designed to assess code generation capabilities in Python. Each problem includes a function signature and several exam-ples that specify the intended functionality. (2) MBPP (Austin et al., 2021) is a large-scale dataset containing short, human-written Python programming problems covering diverse skills. Each problem includes a function signature and one example test case sampled from the dataset. (3) MBPP Pro (Yu et al., 2024) is an expanded versions of the MBPP benchmarks. Consider-ing the issue of incorrect answer labeling in the MBPP dataset that hinder performance improve-ment, we consider the more challenging MBPP Pro dataset to validate the method’s performance. For fair comparison, all test cases are held out during system execution and are only used for post-hoc correctness verification. For both datasets, the Bash tool is enabled for running LLM-generated test cases and analyzing execu-tion results. Pass@1 is adopted as the evaluation metric.
