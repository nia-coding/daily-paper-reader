## 摘要
我们推出了 ISO-Bench，这是一个旨在测试编程智能体在现实世界推理优化任务中能力的基准测试。这些任务源自 vLLM 和 SGLang 这两个最流行的 LLM 服务框架。每个任务都为智能体提供代码库和瓶颈描述，智能体需生成优化补丁，并与人类专家的解决方案进行对比评估。我们从已合并且具有显著性能提升的拉取请求（Pull Request）中精选了 54 个任务。现有的基准测试多采用基于运行时的指标，但此类方法可能被投机取巧以通过测试，却未能捕捉代码修改的真实意图。因此，我们结合了硬性（基于执行）和软性（基于 LLM）指标，证明两者对于全面评估均不可或缺。通过对闭源和开源编程智能体的评估，我们发现没有单一智能体能在不同代码库中均表现卓越。令人惊讶的是，智能体往往能准确识别瓶颈，却无法实现有效的解决方案。我们还发现，基于相同底层模型的智能体表现差异巨大，这表明脚手架（scaffolding）与模型本身同样重要。

## Abstract
We introduce ISO-Bench, a benchmark for coding agents to test their capabilities on real-world inference optimization tasks. These tasks were taken from vLLM and SGLang, two of the most popular LLM serving frameworks. Each task provides an agent with a codebase and bottleneck description, whereby the agent must produce an optimization patch evaluated against expert human solutions. We curated 54 tasks from merged pull requests with measurable performance improvements. While existing benchmarks heavily use runtime-based metrics, such approaches can be gamed to pass tests without capturing the actual intent of the code changes. Therefore, we combine both hard (execution-based) and soft (LLM-based) metrics to show that both are necessary for complete evaluation. While evaluating both closed and open-source coding agents, we find no single agent dominates across codebases. Surprisingly, agents often identify correct bottlenecks but fail to execute working solutions. We also show that agents with identical underlying models differ substantially, suggesting scaffolding is as important as the model.

#### 1. 核心问题与研究动机
*   **研究背景**：随着大语言模型（LLM）推理引擎（如 vLLM 和 SGLang）在生产环境中的广泛应用，系统级优化（如内存管理、内核开发和调度）变得至关重要。
*   **核心问题**：现有的编程智能体（Coding Agents）虽然在修复 Bug（如 SWE-bench）方面表现出色，但在**性能优化**任务上仍面临巨大挑战。
*   **动机**：现有的性能基准测试（如 SWE-Perf, GSO）主要依赖运行时指标（硬指标），这可能导致智能体通过“投机取巧”（Lucky Wins）达成目标，而未真正理解优化意图。论文旨在构建一个结合执行性能与代码意图理解的深度评估基准。

#### 2. 方法论：ISO-Bench 框架
*   **核心思想**：通过“硬指标”（执行结果）和“软指标”（意图分析）的双重评估，将智能体的表现划分为四个象限，以区分真正的成功与偶然的改进。
*   **任务构建流程**：
    1.  **自动筛选**：从 vLLM 和 SGLang 的 GitHub 提交记录中，利用关键词过滤和 LLM 分类，提取与 GPU 推理优化相关的 PR。
    2.  **人工精选**：验证优化的可复现性，提取基准测试命令，并编写不含答案的瓶颈描述。
*   **评估体系（四象限框架）**：
    *   **Q1 (True Success)**：目标正确且性能提升。
    *   **Q2 (Good Intent, Bad Execution)**：目标正确但实现失败（代码报错或性能下降）。
    *   **Q3 (Lucky Win)**：目标错误但性能意外提升（可能破坏了正确性）。
    *   **Q4 (Complete Failure)**：目标错误且性能未提升。
*   **指标定义**：
    *   **硬指标**：首字延迟（TTFT）变化率和吞吐量（Throughput）变化率。
    *   **软指标**：使用 LLM-as-a-Judge（如 Gemini-3-Flash）对比智能体补丁与人类专家补丁的“瓶颈定位”和“实现方法”。

#### 3. 实验设计
*   **数据集/场景**：共 54 个优化任务（39 个来自 vLLM，15 个来自 SGLang）。
*   **对比方法（智能体脚手架）**：
    *   **Claude Code**：Anthropic 的专有智能体（基于 Claude Sonnet 4.5）。
    *   **Codex CLI**：OpenAI 的命令行工具（基于 GPT-5）。
    *   **TRAE-Agent**：字节跳动开源框架的修改版，分别搭载 Claude Sonnet 4.5 和 GPT-5 进行测试。
*   **基准对比**：以人类专家的原始 PR 补丁作为性能和逻辑的基准（Ground Truth）。

#### 4. 资源与算力
*   **硬件环境**：每个任务在独立的 Docker 容器中运行，配备 **1 张 NVIDIA H100 GPU (80GB VRAM)**。
*   **算力分配**：每个智能体针对每个任务有 **120 分钟** 的尝试时间，期间允许自由探索代码库、运行测试和迭代补丁。
*   **辅助资源**：容器限制为 8 核 CPU 和 64GB RAM。

#### 5. 实验数量与充分性
*   **实验规模**：针对 4 种智能体配置在 54 个任务上进行了完整测试，并对所有“硬指标成功”的案例进行了功能正确性验证（使用 LM Evaluation Harness）。
*   **充分性评价**：
    *   **客观性**：通过跨代码库（vLLM vs SGLang）的对比，揭示了智能体表现的非泛化性。
    *   **公平性**：所有智能体使用相同的任务提示词和资源限制。
    *   **深度分析**：不仅关注成功率，还深入分析了开源模型（如 GLM-4.7, MiniMax-M2.1）的失败模式（如死循环、环境混淆等）。

#### 6. 主要结论与发现
*   **理解与执行的鸿沟**：智能体经常能准确识别性能瓶颈（Q2 占比高），但往往无法写出可运行或有效的优化代码。
*   **硬指标的误导性**：仅靠执行指标会使性能评估虚高 **10%-20%**，因为存在“幸运赢家”（Q3），这些补丁有时通过牺牲模型精度（如硬编码维度）来换取速度。
*   **脚手架的重要性**：使用相同底层模型（Sonnet 4.5）的 Claude Code 和 TRAE-Agent 在不同项目上的表现截然不同，证明智能体框架的设计与模型本身同样重要。
*   **无绝对霸主**：没有一个智能体能在两个代码库中同时保持领先，表现具有高度的项目依赖性。

#### 7. 优点
*   **现实意义强**：任务直接源自顶级开源推理框架的真实优化案例，而非合成题目。
*   **评估维度全面**：引入软指标和四象限分析，能够有效识别“奖励黑客”（Reward Hacking）行为。
*   **功能验证严谨**：对优化后的代码进行了精度测试，确保性能提升不是以破坏模型输出为代价。

#### 8. 不足与局限
*   **样本量有限**：仅覆盖了两个代码库和 54 个任务，且排除了涉及 10 个文件以上的大规模系统重构。
*   **数据污染风险**：任务源自公开 PR，先进模型在训练阶段可能已经见过这些代码。
*   **硬件局限**：目前仅限于单卡 H100 测试，未涉及多卡并行（张量并行、流水线并行）等更复杂的生产优化场景。
*   **软指标依赖**：软指标评估依赖于单一的 LLM 裁判，可能存在主观偏差。
