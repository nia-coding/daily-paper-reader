## 摘要
当前的图形用户界面（GUI）智能体主要在反应式范式下运行：用户必须提供明确的指令，智能体才能执行任务。然而，智能 AI 助手应该是主动的，能够直接从连续的视觉输入（如移动端或桌面端截图）中预测用户意图，并在没有用户明确提示的情况下提供及时的建议。转向这种主动范式面临着重大挑战。现实世界的屏幕活动很少是线性的；它由长程轨迹组成，充满了噪声浏览、无意义的操作以及多线程任务切换。为了填补这一空白，我们推出了 PIRA-Bench（主动意图推荐智能体基准测试），这是一个用于在连续、弱监督视觉输入上评估多模态大语言模型（MLLM）的新型基准。与反应式数据集不同，PIRA-Bench 的特点是具有多个交织意图的复杂轨迹，以及带有各种用户画像背景的噪声片段，挑战智能体在适应用户偏好的同时检测可执行事件。此外，我们提出了 PIRF 基准模型，这是一个具备记忆感知和状态追踪能力的框架，使通用 MLLM 能够管理多个任务线程并处理具有误导性的视觉输入。PIRA-Bench 是迈向稳健且主动的基于 GUI 的个人助手的第一步。

## Abstract
Current Graphical User Interface (GUI) agents operate primarily under a reactive paradigm: a user must provide an explicit instruction for the agent to execute a task. However, an intelligent AI assistant should be proactive, which is capable of anticipating user intentions directly from continuous visual inputs, such as mobile or desktop screenshots, and offering timely recommendations without explicit user prompting. Transitioning to this proactive paradigm presents significant challenges. Real-world screen activity is rarely linear; it consists of long-horizon trajectories fraught with noisy browsing, meaningless actions, and multithreaded task-switching. To address this gap, we introduce PIRA-Bench (Proactive Intent Recommendation Agent Benchmark), a novel benchmark for evaluating multimodal large language models (MLLMs) on continuous, weakly-supervised visual inputs. Unlike reactive datasets, PIRA-Bench features complex trajectories with multiple interleaved intents and noisy segments with various user profile contexts, challenging agents to detect actionable events while fitting to user preferences. Furthermore, we propose the PIRF baseline, a memory-aware, state-tracking framework that empowers general MLLMs to manage multiple task threads and handle misleading visual inputs. PIRA-Bench serves as an initial step toward robust and proactive GUI-based personal assistants.

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题**：当前的 GUI 智能体（如 UI-TARS, Mobile-Agent 等）主要遵循**反应式（Reactive）**范式，即必须依赖用户提供明确、详尽的指令才能执行任务。这增加了用户的认知负担，且在用户忘记细节或多任务并行时表现不佳。
*   **研究动机**：真正的 AI 助手应当是**主动式（Proactive）**的。它应能通过持续监测屏幕视觉流（截图），在用户下令前识别其潜在意图（Latent Intent），并主动推荐可执行的操作（如：根据聊天内容主动建议订餐厅或创建日程）。
*   **挑战**：现实中的屏幕活动具有非线性、长程、充满噪声（如无意义的滑动、App切换）以及多任务交织（Multithreaded task-switching）等特点，使得从视觉流中精准提取意图极具挑战。

### 2. 论文提出的方法论
论文提出了一个评估基准 **PIRA-Bench** 和一个基线框架 **PIRF**。

#### **核心思想：主动意图推荐（PIR）**
将任务建模为从视觉轨迹 $T$ 和用户画像 $P$ 到意图集合 $I^*$ 的映射：$I^* = \arg \max P_\theta(I | T, P)$。

#### **关键技术细节：PIRF 框架**
为了解决长程上下文丢失和幻觉问题，PIRF 采用了以下设计：
*   **动态记忆模块（Dynamic Memory Module）**：
    *   **静态部分**：存储用户画像（经济地位、偏好等）。
    *   **动态部分**：维护一个“意图银行”，记录多个并行的任务线程。
*   **状态转移机制（Action Space）**：模型在处理每一帧时需输出四种状态之一：
    *   `CREATE`：发现新任务。
    *   `RESUME`：切换回之前的任务。
    *   `UPDATE`：更新当前任务进度。
    *   `IDLE`：识别为噪声，不采取行动。
*   **反射与自动删除机制（Reflection & Auto-Deletion）**：模型会持续评估记忆中的意图是否已完成或被用户放弃，并自动清理过期意图，以减少上下文冗余和幻觉。
*   **滑动窗口**：仅保留最近的 $K=10$ 帧图像作为即时视觉输入。

### 3. 实验设计
*   **数据集 (PIRA-Bench)**：
    *   包含 **100 条** 真实世界的 GUI 轨迹，平均每条包含 33 张截图。
    *   **多维场景**：涵盖直接推荐（仅靠视觉）、画像依赖（需结合用户身份判断意图）和纯噪声拒绝（测试模型是否会乱猜）。
    *   **用户画像**：每条轨迹配有 3 种不同的用户画像（如学生 vs. 富裕人士），导致不同的意图预测结果。
*   **评估指标**：
    *   **$F1_{avg}$**：正向样本的意图预测准确率与召回率的综合。
    *   **$FPS_{norm}$**：归一化误报得分，惩罚在噪声片段下的“过度主动”（幻觉）。
    *   **$S_{final}$**：最终得分，$S_{final} = F1_{avg} \times FPS_{norm}$。
*   **对比方法**：
    *   **模型**：Gemini-3.1-Pro, GPT-5.2, Qwen3.5-Plus, Seed-1.8。
    *   **设置**：Naive MLLM（简单滑动窗口） vs. PIRF（带记忆和状态追踪）。
    *   **基准**：人类表现（Human Performance）作为上限参考。

### 6. 论文的主要结论与发现
*   **PIRF 的有效性**：在所有模型上，PIRF 框架均显著优于 Naive 模式，证明了状态追踪和记忆管理在长程主动任务中的必要性。
*   **“过度主动”陷阱**：强模型（如 GPT-5.2）虽然召回率高，但在噪声环境下极易产生幻觉（Precision 仅约 50%），导致其在真实场景中可能因频繁打扰用户而不可用。
*   **保守策略的优势**：Seed-1.8 在 $S_{final}$ 上表现最好，因为它在噪声抑制（$FPS_{norm}$）方面更保守，虽然漏掉了一些意图，但错误更少。
*   **巨大的鸿沟**：人类在 $S_{final}$ 上得分约 90%，而表现最好的模型仅为 28%，说明 MLLM 在区分“真实意图”与“随机浏览”方面仍有极长路要走。

### 7. 优点（亮点）
*   **范式创新**：率先提出了从“反应式”向“主动式”GUI 智能体转变的概念，符合未来 AI 助手的演进方向。
*   **基准设计科学**：PIRA-Bench 不仅关注任务完成，还引入了**用户画像依赖**和**噪声拒绝**，极大地贴近了真实使用环境。
*   **评估指标合理**：通过乘法结合 $F1$ 和 $FPS$，强制模型在“能力”与“克制”之间寻找平衡。

### 8. 不足与局限
*   **数据集规模**：100 条轨迹虽然质量高，但样本量较小，可能无法完全覆盖复杂的 App 交互组合。
*   **LLM-as-a-Judge 风险**：评估依赖 Gemini-3-flash 进行语义匹配，可能存在模型偏见或对细微意图理解不准的问题。
*   **实时性挑战**：PIRF 需要在每一帧或每隔几帧进行推理和记忆更新，在移动设备上的端侧部署可能会面临严重的延迟和功耗挑战。
*   **应用限制**：目前仅关注意图“推荐”，尚未涉及推荐后的自动化执行及用户反馈闭环。
